# Требования к Android VPN-клиенту (на базе Xray-core)

**Версия документа:** 0.1 (черновик)
**Тип продукта:** Android-клиент к прокси-серверам Xray (аналог v2rayNG / OneXray)
**Статус:** согласование требований перед этапом дизайна

---

## 1. Обзор и цель

Разрабатывается **клиентское** Android-приложение, которое подключается к прокси-серверам по протоколам Xray-core. Приложение **не является VPN-сервисом** и не содержит собственных серверов — оно управляет соединениями к серверам, которые предоставляет пользователь или оператор.

Отправная точка проекта — клиент под **собственный VPN-сервис** оператора, но продукт делается как **универсальный** Xray-клиент, в котором собственный сервис выступает «дружелюбным дефолтом».

### 1.1 Ключевые продуктовые решения

| Решение | Выбор | Обоснование |
|---|---|---|
| Ядро | **Xray-core** (через `libXray`, gomobile-биндинг) | Сервер использует VLESS + Reality/Vision — Xray-core эталонная реализация этих фич |
| Тип клиента | Универсальный + свой сервис как дефолт | Пригоден для F-Droid как самостоятельный инструмент; простой онбординг новичкам |
| Туннель | Android `VpnService` (TUN) + tun2socks | Стандартный подход для перехвата системного трафика |
| Распространение | GitHub (APK) + F-Droid | Без ограничений модерации Google Play |
| Лицензирование | 100% открытый код (FOSS), без проприетарных SDK | Требование F-Droid |
| Мин. версия ОС | Android 8.0 (API 26) | Покрытие ~99% активных устройств, стабильный `VpnService` |
| Целевая версия | Последний стабильный API | Требования Play не применяются, но держим актуальность |

## 2. Целевая аудитория и сценарии

- **Новичок оператора сервиса.** Ставит приложение, в один тап подключается к предустановленному сервису (вшитая подписка / QR), не разбираясь в настройках.
- **Продвинутый пользователь.** Добавляет свои серверы (VLESS/Reality, VMess, Trojan, Shadowsocks) вручную, через QR или ссылку-подписку.
- **Пользователь в среде фильтрации трафика.** Нуждается в устойчивом соединении (Reality/Vision) и в том, чтобы часть приложений (банки, локальные сервисы) ходила напрямую.

Основной сценарий MVP: *открыть → импортировать конфиг → выбрать сервер → подключиться → (опц.) отметить приложения для обхода.*

---

## 3. Архитектура (высокоуровнево)

- **UI-слой:** Kotlin + Jetpack Compose, паттерн MVI (однонаправленный поток данных: Intent → State → View).
- **Ядро:** Xray-core, собранное как нативная библиотека через `libXray`/gomobile, вызовы через JNI.
- **Сеть:** `VpnService` создаёт TUN-интерфейс; tun2socks (напр. hev-socks5-tunnel) заворачивает TUN → локальный SOCKS-инбаунд Xray.
- **Хранилище:** локальная БД (Room) для серверов/подписок; чувствительные поля шифруются (Android Keystore / EncryptedSharedPreferences).
- **Foreground service:** соединение живёт в foreground-сервисе с постоянным уведомлением (иначе система убивает процесс).

## 4. Область релизов

### 4.1 MVP (первый релиз)

Обязательный минимум:

1. Подключение / отключение (кнопка + статус).
2. Импорт конфигурации: ссылка/подписка (URL), QR (скан и распознавание из изображения), ручной ввод.
3. Список серверов: добавление, удаление, редактирование, выбор активного, тест задержки (ping).
4. **Per-app split tunneling (режим обхода):** экран со списком установленных приложений; отмеченные приложения ходят **напрямую**, минуя прокси.
5. Дружелюбный дефолт: предложение подключиться к предустановленному сервису оператора.

### 4.2 Roadmap (последующие релизы)

- Правила роутинга (geoip/geosite, по домену/IP), пресеты «через прокси только заблокированное».
- Блокировка рекламы (geosite-категории).
- Kill switch / Always-on VPN.
- Quick Settings-плитка + автозапуск при загрузке.
- Авто-обновление подписок, группы серверов, спид-тест, «подключиться к быстрейшему».
- Экран трафика/квоты (если у сервиса есть учёт).
- Бэкап/экспорт профилей.
- (Стратегически) второе ядро sing-box для Hysteria2/TUIC.

## 5. Функциональные требования

Приоритеты: **M** — MVP, **L** — later.

### 5.1 Протоколы и транспорты

- **FR-01 (M):** Поддержка **VLESS** с Reality, XTLS-Vision (`xtls-rprx-vision`), TLS.
- **FR-02 (M):** Поддержка VMess, Trojan, Shadowsocks, SOCKS/HTTP (универсальность).
- **FR-03 (M):** Транспорты: TCP, WebSocket, gRPC, HTTPUpgrade/xHTTP, mKCP, Reality.

### 5.2 Импорт и управление конфигами

- **FR-10 (M):** Импорт по ссылке-подписке (base64/URL), с ручным обновлением.
- **FR-11 (M):** Импорт по QR — скан камерой и распознавание из изображения галереи.
- **FR-12 (M):** Ручное создание/редактирование сервера через форму.
- **FR-13 (M):** Импорт из буфера обмена.
- **FR-14 (L):** Импорт из файла и по deep-link (URI-схема приложения).
- **FR-15 (L):** Несколько подписок с авто-обновлением по расписанию.

### 5.3 Соединение

- **FR-20 (M):** Установка/разрыв соединения через `VpnService`; запрос системного разрешения на VPN при первом запуске.
- **FR-21 (M):** Индикация статуса: отключено / подключение / подключено / ошибка.
- **FR-22 (M):** Foreground-сервис + постоянное уведомление с быстрым отключением.
- **FR-23 (M):** Отображение текущей скорости и переданного объёма трафика (базово).
- **FR-24 (M):** Тест задержки (ping) до серверов в списке.

### 5.4 Серверы

- **FR-30 (M):** CRUD серверов, выбор активного.
- **FR-31 (M):** Отображение метаданных (протокол, адрес, задержка).
- **FR-32 (L):** Группировка, сортировка, «подключиться к быстрейшему».

### 5.5 Per-app split tunneling (обход)

- **FR-40 (M):** Экран со списком установленных приложений (иконка, имя, package).
- **FR-41 (M):** Отметка приложений, чей трафик идёт **напрямую** (реализация через `addDisallowedApplication`).
- **FR-42 (M):** Сохранение выбора; применение при следующем старте соединения (с подсказкой перезапустить VPN, если оно активно).
- **FR-43 (L):** Инверсный режим — «через прокси только выбранные приложения» (`addAllowedApplication`).

### 5.6 Свой сервис (дружелюбный дефолт)

- **FR-50 (M):** На онбординге — кнопка «Подключиться к [сервис оператора]» по предустановленной подписке/QR.
- **FR-51 (M):** «Добавить свой сервер» доступно рядом, без скрытия за настройками.

## 6. Нефункциональные требования

- **NFR-01 (Приватность):** Никакой телеметрии и аналитики; логи трафика не собираются и не хранятся. Диагностические логи — только локально и по явному действию пользователя.
- **NFR-02 (Безопасность):** Конфиги и учётные данные шифруются на диске (Keystore); нет передачи данных в открытом виде наружу приложения.
- **NFR-03 (F-Droid):** Только свободные зависимости; Go-ядро собирается из исходников; цель — воспроизводимые сборки. Валидировать сборку Xray-core в пайплайне F-Droid.
- **NFR-04 (Производительность):** Минимальные накладные расходы CPU/батареи в простое; стабильное удержание соединения в фоне.
- **NFR-05 (Совместимость):** Android 8.0–15; корректная работа на «агрессивных» вендорах (Xiaomi/Huawei/Samsung) — подсказка на отключение оптимизации батареи.
- **NFR-06 (Локализация):** i18n с первого дня; минимум RU + EN (аудитория обхода многоязычна — задел на FA/ZH).
- **NFR-07 (Доступность):** Базовая a11y (контраст, размеры целей нажатия, TalkBack).
- **NFR-08 (Обновляемость):** Механизм обновления geosite/geoip и ядра без полной пересборки, где возможно (гонка вооружений при обходе).

## 7. UX / экраны MVP

1. **Онбординг** — приветствие, кнопка «Подключиться к сервису» + «Добавить свой сервер», запрос разрешения VPN.
2. **Главный** — большая кнопка вкл/выкл, статус, активный сервер, счётчик трафика/скорости.
3. **Серверы** — список, добавление (+), импорт (URL/QR/буфер/ручной), ping, выбор активного.
4. **Импорт** — сканер QR, поле ссылки, форма ручного ввода.
5. **Split tunneling** — список приложений с чекбоксами (обход).
6. **Настройки** — базовые (тема, язык, ссылка на подписку сервиса, диагностика).

## 8. Ограничения и риски

- **Синхронизация Reality/Vision.** Версии Xray-core на клиенте и сервере должны быть совместимы; расхождение ломает соединение. → зафиксировать процесс обновления ядра.
- **F-Droid + Go-ядро.** Воспроизводимая сборка Xray-core в пайплайне F-Droid — нетривиальна; проверить на раннем этапе (риск для срока публикации).
- **Гонка вооружений.** Методы фильтрации меняются; нужен быстрый канал доставки обновлений ядра/правил.
- **Фоновые ограничения вендоров.** Убийство процессов; смягчается foreground-сервисом и запросом исключения из оптимизации батареи.
- **Лицензии.** Xray-core — MPL 2.0. При заимствовании кода из v2rayNG (GPLv3) весь проект попадает под GPLv3 — учесть при выборе, писать ли с нуля или форкать. *(Не юридическая консультация — уточнить отдельно.)*

## 10. Глоссарий

- **Xray-core** — прокси-движок семейства V2Ray, реализующий VLESS/VMess/Trojan/SS и Reality/Vision.
- **Reality / XTLS-Vision** — методы маскировки трафика под настоящий TLS для устойчивости к фильтрации.
- **VpnService** — Android-API для создания TUN-интерфейса и перехвата системного трафика.
- **tun2socks** — компонент, преобразующий пакеты TUN в SOCKS-поток для ядра.
- **Split tunneling / per-app proxy** — маршрутизация трафика отдельных приложений мимо прокси (обход) или, наоборот, только через прокси.
- **libXray** — мобильный биндинг Xray-core для вызова из Android/iOS.
